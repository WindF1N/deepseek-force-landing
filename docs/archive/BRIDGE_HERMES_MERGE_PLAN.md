# План слияния DeepSeek Bridge и Hermes Agent

## Версия 3.0 — Полная мультиагентная архитектура с пулом сессий и оркестратором

---

## 1. Цель

Создать единую систему, в которой **Hermes Agent** является основным интерфейсом для пользователя, а **DeepSeek Bridge** выступает в роли прозрачного прокси к веб-версии DeepSeek с поддержкой **параллельных сессий**. Вся интеллектуальная нагрузка (планирование, декомпозиция, параллелизация, валидация, агрегация) вынесена в отдельный слой — **Оркестратор**.

**Ключевые особенности:**
- Bridge управляет **пулом сессий DeepSeek**, каждая сессия = одно Chrome-окно/вкладка.
- Оркестратор может отправлять параллельные запросы в Bridge, используя разные сессии.
- Мультиагентный подход: один запрос от Hermes → множество параллельных/последовательных агентов DeepSeek.
- **DeepSeek не генерирует большие блоки кода** — он только планирует действия, а Оркестратор их выполняет.
- **JSON-парсинг вынесен в Оркестратор**, где используется `pydantic` и `jsonschema`.

## 3. Компоненты системы

### 3.1. Оркестратор (новый слой)

Оркестратор — это отдельный сервис (или модуль), который принимает запросы от Hermes по OpenAI-совместимому API и обрабатывает их с использованием мультиагентного подхода.

#### 3.1.1. Planner (планировщик)

**Задача:** Разбить большой запрос на логические шаги.

**Вход:**
- Текст запроса (`messages`).
- Список доступных инструментов (`tools`).

**Выход:**
- JSON-план: `[{"step": 1, "description": "...", "tools": [...], "depends_on": []}]`.

**Реализация:**
- Использует **один быстрый вызов DeepSeek (Expert)** с промтом:
  ```
  Ты — планировщик. Разбей эту задачу на 3–7 логических шагов.
  Для каждого шага укажи:
  - Описание шага (кратко).
  - Какие инструменты нужны (из списка [TOOLS]).
  - Зависит ли от других шагов (depends_on).
  Верни JSON: [{"step": 1, "description": "...", "tools": [...], "depends_on": []}]
  ```
- Если задача простая (1–2 шага), планировщик может пропустить декомпозицию и сразу передать задачу в Executor.

#### 3.1.2. Executor (исполнитель)

**Задача:** Выполнить шаги плана.

**Вход:**
- План (JSON от Planner).
- Контекст (история, файлы).

**Выход:**
- Результаты выполнения каждого шага.

**Режимы выполнения:**
| Режим | Описание | Когда использовать |
| :--- | :--- | :--- |
| **Sequential** | Шаги выполняются последовательно, результат каждого передаётся следующему. | Когда задача линейная (напиши план → напиши код → проверь код). |
| **Parallel** | Шаги выполняются параллельно (через `asyncio.gather`). | Когда шаги независимы (проанализируй 5 файлов одновременно). |
| **Hybrid** | Комбинация: параллельные группы, внутри которых последовательные шаги. | Для сложных задач (проверь 3 репозитория → для каждого напиши документацию → собери общий отчёт). |

**Управление сессиями:**
- Каждый агент (вызов DeepSeek) имеет свой `chat_session_id` и `parent_message_id`.
- После выполнения шага сессия может быть закрыта или сохранена для будущих запросов.
- Лимит параллельных сессий: настраивается в `config.py` (по умолчанию 5–10).

#### 3.1.3. Validator (валидатор)

**Задача:** Проверить ответы DeepSeek на корректность.

**Вход:**
- Ответ DeepSeek (содержимое).
- Ожидаемый формат (JSON-схема).

**Выход:**
- `True` — если ответ валидный.
- `False` + сообщение об ошибке — если невалидный.

**Реализация:**
- Использует `pydantic` и `jsonschema` для валидации.
- Проверяет:
  - Валидный ли JSON?
  - Есть ли все обязательные поля?
  - Корректны ли типы данных?
  - Не противоречит ли результат предыдущим шагам?

**Recovery-механизм:**
- Если ответ невалидный, отправляет **короткий recovery-промпт**:
  ```
  Ошибка в JSON. Верни исправленный JSON для шага X.
  Только JSON, без пояснений.
  ```
- Максимум 3 попытки recovery на один шаг.
- Если после 3 попыток ответ всё ещё невалидный — шаг пропускается, в лог пишется ошибка.

#### 3.1.4. Aggregator (агрегатор)

**Задача:** Собрать результаты всех шагов и сформировать итоговый ответ.

**Вход:**
- Результаты выполнения каждого шага.
- Исходный запрос пользователя.

**Выход:**
- Стандартный `ChatCompletionResponse` для Hermes.

**Реализация:**
- Если шаги выполнялись последовательно — объединяет их в единый ответ.
- Если шаги выполнялись параллельно — собирает данные в структурированный JSON.
- Добавляет метаданные: количество шагов, время выполнения, использованные инструменты.

### 3.2. DeepSeek Bridge (прокси с пулом сессий)

Bridge становится **прозрачным прокси** для веб-версии DeepSeek с поддержкой **параллельных сессий**.

#### 3.2.1. Что остаётся в Bridge
- `/v1/chat/completions` — основной эндпоинт.
- `/v1/models` — список моделей.
- `/health` — проверка статуса.
- WebSocket (`/ws`) — для связи с расширением.
- Заглушки для совместимости (`/props`, `/version`, `/api/show`).

#### 3.2.2. Что добавляется в Bridge
- **Session Pool Manager**: управляет пулом сессий DeepSeek.
- **Балансировка нагрузки**: распределяет запросы между сессиями (round-robin или least-connections).
- **Очередь запросов**: если все сессии заняты, запросы становятся в очередь.
- **Тайм-ауты**: сессии, которые не используются > X минут, закрываются.

#### 3.2.3. Что удаляется из Bridge
- Telegram-бот (`telegram_bot/`).
- `orchestrator.py` (классификация, контекст, история).
- `summarizer.py`.
- `models.py` (модели БД).
- Вся логика обработки сообщений.
- **Парсинг JSON** (теперь это ответственность Оркестратора).
- **Модификация промтов** (промты передаются как есть).

#### 3.2.4. Промты в Bridge
- Bridge не модифицирует промты.
- Он передаёт их как есть, без добавления маркеров или инструкций.
- Вся интеллектуальная нагрузка — на Оркестраторе.

### 3.3. Chrome Extension (несколько экземпляров)

Расширение теперь должно поддерживать **несколько экземпляров**, каждый из которых создаёт отдельную сессию DeepSeek.

#### 3.3.1. Как это работает
1. При запуске расширение создаёт новую сессию DeepSeek (`createChatSession`).
2. Расширение подключается к Bridge с параметром `session_id`:
   `ws://localhost:8001/ws?key=...&session_id=<uuid>`
3. Bridge сохраняет это подключение как отдельную сессию в пуле.
4. При получении запроса Bridge выбирает свободную сессию и отправляет запрос в соответствующее расширение.
5. После завершения запроса расширение отправляет сигнал `READY`, и сессия снова становится доступной.

#### 3.3.2. Как запустить несколько экземпляров
- **Вариант 1:** Несколько профилей Chrome (каждый со своим расширением).
- **Вариант 2:** Несколько браузеров (Chrome, Brave, Edge).
- **Вариант 3:** Использовать `puppeteer` или `playwright` для управления несколькими окнами.

#### 3.3.3. Что нужно изменить в `content.js`
- Добавить генерацию `session_id` при первом запуске.
- Сохранять `session_id` в `localStorage`.
- Подключаться к Bridge с параметром `session_id`.
- Отправлять сигнал `READY` после завершения каждого запроса.

### 3.4. Hermes Agent (главный интерфейс)

Hermes остаётся основным интерфейсом для пользователя.

**Что меняется:**
- Hermes отправляет запросы не напрямую в Bridge, а в **Оркестратор**.
- Конфиг Hermes обновляется:
  ```yaml
  model:
    default: deepseek-chat
    provider: custom:orchestrator
    base_url: http://localhost:8002/v1  # порт Оркестратора
    api_key: supersecretkey
  ```

**Что остаётся без изменений:**
- Шлюзы (Telegram, Discord, Slack).
- Инструменты (terminal, read_file, write_file, browser, cron, memory).
- Навыки (skills).
- Память (memory).

## 4. Поток выполнения (пример)

**Задача:** «Проанализируй код в папке `/project` и напиши документацию».

### 4.1. Hermes получает запрос
```json
{
  "model": "deepseek-chat",
  "messages": [{"role": "user", "content": "Проанализируй код в папке /project и напиши документацию"}],
  "tools": [...]
}
```

### 4.2. Оркестратор (Planner)
- Шаг 1: `read_file` для всех файлов в `/project`.
- Шаг 2: `analyze` каждого файла (параллельно).
- Шаг 3: `write` общей документации.
- Шаг 4: `validate` документации.

### 4.3. Оркестратор (Executor)
- **Шаг 1 (Sequential)**: один агент DeepSeek читает файлы.
- **Шаг 2 (Parallel)**: 3–5 агентов DeepSeek параллельно анализируют файлы.
- **Шаг 3 (Sequential)**: один агент DeepSeek пишет документацию.
- **Шаг 4 (Sequential)**: один агент DeepSeek валидирует документацию.

### 4.4. Bridge (Session Pool)
- Для Шага 1: берёт свободную сессию из пула.
- Для Шага 2: берёт 3–5 свободных сессий параллельно.
- Для Шага 3: берёт свободную сессию.
- Для Шага 4: берёт свободную сессию.

### 4.5. Оркестратор (Validator)
- Проверяет JSON-ответы каждого шага.
- Если ошибка → recovery-запрос.

### 4.6. Оркестратор (Aggregator)
- Собирает результаты.
- Формирует ответ:
  ```json
  {
    "id": "chatcmpl-...",
    "object": "chat.completion",
    "created": 1234567890,
    "model": "deepseek-chat",
    "choices": [{
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "Документация создана: README.md, API.md"
      },
      "finish_reason": "stop"
    }]
  }
  ```

## 5. Техническая реализация

### 5.1. Структура проекта

```
deepseek-api/
├── bridge/
│   ├── main.py              # Очищенный прокси (FastAPI)
│   ├── ws_server.py         # WebSocket для расширения (с SessionPool)
│   ├── database.py          # (опционально)
│   └── logging_config.py    # Логирование
├── orchestrator/
│   ├── api.py               # FastAPI-эндпоинты
│   ├── planner.py           # Декомпозиция задач
│   ├── executor.py          # Управление выполнением шагов
│   ├── validator.py         # Валидация и recovery
│   ├── aggregator.py        # Сборка результатов
│   ├── session_manager.py   # Управление сессиями DeepSeek
│   ├── bridge_client.py     # Клиент для DeepSeek Bridge
│   └── config.py            # Настройки (таймауты, лимиты)
├── chrome-extension/        # (без изменений, кроме content.js)
├── docker-compose.yml
└── .env
```

### 5.2. Взаимодействие с Hermes

Оркестратор запускается как отдельный сервис (порт 8002).
Hermes отправляет запросы на `/v1/chat/completions` Оркестратора.
Оркестратор внутри использует Bridge как `provider`.

### 5.3. Параллелизация

- Используется `asyncio.gather` для параллельных вызовов Bridge.
- Каждый вызов — отдельная сессия DeepSeek.
- Лимит параллельных сессий: 5–10 (настраивается в `config.py`).

### 5.4. Обработка ошибок

- Если один агент «упал» (невалидный JSON) → его шаг перезапускается (recovery).
- Если агент упал 3 раза → шаг пропускается, в лог пишется ошибка.
- Если критический шаг упал → вся задача останавливается, возвращается ошибка Hermes.

### 5.5. Логирование

- Все компоненты пишут логи в папку `logs/`.
- Уровень DEBUG для детальной отладки.
- Ротация: 5 файлов по 10 МБ.

## 6. Расширение Bridge для поддержки параллельных сессий

### 6.1. Текущая проблема

Сейчас `content.js` работает с **единственным WebSocket-соединением** и создаёт **одну сессию DeepSeek** для каждого запроса. Это означает, что параллельные запросы от Оркестратора будут конкурировать за одно и то же окно/сессию.

### 6.2. Решение: Session Pool Manager

Вместо одного `extension_manager` с одним WebSocket, мы создадим **пул менеджеров**, каждый из которых управляет отдельной сессией DeepSeek (отдельное Chrome-окно/вкладка).

**Структура:**
```
SessionManager:
  - session_id (уникальный)
  - websocket (подключение к расширению)
  - is_busy (флаг: занята ли сессия)
  - created_at, last_used_at

SessionPool:
  - sessions: list[SessionManager]
  - get_available_session() -> SessionManager (или создаёт новую)
  - release_session(session_id)
  - max_sessions (настраиваемый лимит)
```

### 6.3. Изменение логики `extension_manager`

Сейчас `extension_manager` — это синглтон с одним списком `connections`. Нужно переписать его на `SessionPool`
