# Roadmap: DeepSeek Bridge-Orchestrator как публичный API-сервис

Документ фиксирует план превращения локального бриджа в удалённый
многопользовательский API-эндпоинт. Сохранено по ходу обсуждения, чтобы не
терять контекст между сессиями.

---

## 1. Цель пользователя

Развернуть на сервере удалённый OpenAI-совместимый API-эндпоинт, к которому
подключаются ~200 человек. На сервере запущено много профилей Google Chrome,
каждый залогинен в свой аккаунт DeepSeek (chat.deepseek.com). Все эти сессии
используются как пул для обслуживания запросов пользователей.

Основной сценарий использования: **продаём связку как OpenAI-совместимый
провайдер для Hermes Agent** (агентный tool-calling). 200 человек
подключают свои Hermes к нашему эндпоинту и получают DeepSeek как "мозг"
с инструментами. Это сценарий А (агентный) — НЕ чат для людей.

---

## 2. Математика сессий (грубая прикидка)

Базовая величина: один ход ≈ 8–12 c (создание chat_session + PoW + генерация).
Берём 10 c → одна сессия ≈ 6 запросов/мин.

| Сценарий | Оценка нужных сессий (профилей Chrome) |
|----------|----------------------------------------|
| Лёгкий чат (1 запрос / 2 мин на юзера), 200 чел | ~17 базово, ×2 на пики = **30–50** |
| То же + голосование VOTE_N=4 | ×4 = **120–160** |
| Агентный tool-loop (диалог держит сессию минутами) | **200+**, с голосованием — нереально (≈800) |

Вывод: для лёгкого чата/кода 30–50 профилей реалистично. Голосование и
агентный режим резко умножают потребность — для публичного сервиса на людей
их, скорее всего, отключить.

**КРИТИЧЕСКОЕ НЕИЗВЕСТНОЕ:** сколько запросов выдерживает ОДИН аккаунт
DeepSeek до rate-limit/бана. От этой цифры зависит вся экономика (нужно 30
аккаунтов или 300). Это надо измерить стресс-тестом ДО постройки кластера.

---

## 3. Три стены (важнее числа сессий)

### Стена 1 — баны аккаунтов / IP со стороны DeepSeek
- Каждый профиль Chrome = один аккаунт DeepSeek.
- 200 человек через горстку аккаунтов = аномальный паттерн → rate-limit,
  поднятие сложности PoW, временные баны.
- Сотни сессий с одного серверного IP = красный флаг антибота.
- Нужно: много аккаунтов + ротация + резидентные прокси на каждый профиль.
  Операционно дорого и хрупко.

### Стена 2 — ресурсы сервера на Chrome-профили
- Один профиль с открытой вкладкой ≈ 300–700 МБ RAM + CPU на PoW (WASM-хеши
  — реальная нагрузка на CPU).
- 100 профилей = 30–70 ГБ RAM + постоянный CPU.
- 200–800 профилей не влезут на одну машину → нужен кластер headless-Chrome
  (puppeteer/playwright), оркестрация контейнеров, авто-рестарт.

### Стена 3 — текущая архитектура не готова
Сейчас Bridge держит сессии в памяти ОДНОГО процесса (`self.sessions` dict):
- не переживает рестарт;
- не масштабируется на несколько машин;
- один общий `WS_AUTH_KEY` на всех (нет auth per-user);
- нет rate-limit/учёта/биллинга per-user;
- очередь не переживает перезапуск.

## 4. Целевая архитектура для 200 юзеров

1. **Фронт API** (оркестратор) — несколько реплик за балансировщиком,
   auth per-user (API-ключи), rate-limit на юзера, учёт.
2. **Брокер очереди** (Redis / RabbitMQ) — запросы в очередь, воркеры
   разбирают; переживает рестарт, распределяется между машинами.
3. **Пул воркеров-браузеров** — кластер headless-Chrome на нескольких
   машинах; каждый регистрирует свои сессии в общем реестре.
4. **Реестр сессий в Redis** — состояние free/busy/dead, TTL, health-check,
   авто-пересоздание.
5. **Прокси-слой** — резидентный прокси на профиль, ротация аккаунтов.

Это полноценный распределённый сервис (недели работы + расходы на прокси,
аккаунты, серверы), а не доработка текущего бриджа.

---

## 5. Рекомендованный порядок действий

1. **Стресс-тест одного аккаунта** — найти потолок "запросов/мин до
   бана/429". Самое важное число; либо подтвердит идею, либо сэкономит
   деньги на провале. (Нужен расходный аккаунт под риск.)
2. **Реестр сессий in-memory → Redis** — фундамент для мультимашинного
   масштабирования.
3. **Per-user auth + rate-limit** в оркестраторе.
4. **Headless-Chrome воркеры** (puppeteer) вместо ручных окон — авто-подъём
   профилей.

---

## 6. Режимы DeepSeek — что используется и что нужно

В DeepSeek-вебе 4 независимых рычага. Текущее состояние в коде:

| Рычаг | Что делает | Сейчас в коде | Проблема |
|-------|-----------|---------------|----------|
| `thinking_enabled` | режим рассуждения (DeepThink/R1) | main.py хардкод `True`, НО content.js шлёт `false` | **рассинхрон — флаг не работает** |
| `model_type:"expert"` | Expert (reasoner) для кода/сложного | по "expert" в имени модели | ок |
| `search_enabled` | поиск в интернете | выключен по умолчанию (ломал tool-calls) | ок |
| vision (`x-model-type`) | картинки/файлы | по наличию файлов | ок |

### Решение по режимам (ФИНАЛ — связка для Hermes, агентный tool-calling)

- **Expert — основной режим генерации.** Сильнее для кода/рассуждений,
  Hermes гоняет именно сложные задачи.
- **thinking — ВЫКЛ.** Загрязняет ответ цепочкой мыслей (надо вырезать
  перед парсингом tool_calls) и провоцирует прозу вместо чистого JSON.
  Сейчас фактически выключен (рассинхрон) — это случайно правильно, но
  надо убрать хардкод `True`, чтобы не путал.
- **search — ВЫКЛ.** Ломает формат tool-calls, превращает в поисковик.
- **vision — ВКЛ.** Hermes шлёт картинки/файлы — реальная фича агента.

tool-calling / голосование / планировщик/оркестратор v3.0 — НУЖНЫ, это и
есть продукт (надёжный агентный провайдер).

### План наведения порядка в режимах
- Убрать хардкод `thinking_enabled: True` в main.py (поставить False/опц.).
- Дефолт = expert для агентных запросов.
- Явное управление через имя модели:
  - `deepseek-expert` — основной (по умолчанию для Hermes)
  - `deepseek-chat` — быстрый, для лёгких ходов
  - `deepseek-vision` — файлы/картинки
  - (search/think — только если кто-то явно попросит, не в дефолте)

## 7. Продукт зафиксирован: агентная связка для Hermes

Продаём **OpenAI-совместимый провайдер для Hermes Agent**. Клиент подключает
свой Hermes к нашему эндпоинту и получает DeepSeek-веб как агентный "мозг"
с tool-calling.

Что это требует (всё уже строится):
- expert как основной режим, search/think выключены, vision включён;
- надёжный tool-loop (нативные OpenAI tool_calls, парсинг/валидация);
- голосование/оркестратор — для надёжности формата на сложных ходах;
- масштабируемый пул сессий + обход банов — главная инженерная работа.

Это НЕ чат для людей. Фокус: надёжность агентного формата + масштаб пула.

## Статус на момент записи
- Оркестратор (planner/executor/validator/aggregator) построен, 15/15 тестов
  зелёные.
- Bridge: добавлена честная очередь сессий (asyncio.Condition), атомарный
  захват, retry на пустой ответ. Нагрузка 12 запросов / 4 сессии = 0 ошибок.
- Подключено к Hermes профилем `deepseek`.
- Эти улучшения — под сценарий А. Если финальный выбор — сценарий Б,
  агентную обвязку можно не тащить в прод, но очередь/пул/масштабирование
  нужны в обоих случаях.
