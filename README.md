## Общее описание

**DeepSeek Provider Bridge** — это прокси-сервис, который позволяет взаимодействовать с веб-версией DeepSeek через Telegram-бота, используя живую сессию браузера. Система автоматически выбирает оптимальный режим (Instant с поиском, Expert для сложных задач, Vision для изображений и файлов), поддерживает загрузку до 50 файлов в одном запросе и сохраняет полную историю диалогов в PostgreSQL.

## Архитектура

```
Пользователь (Telegram)
       │
       ▼
┌─────────────────────┐
│   Telegram Bot      │  python-telegram-bot / aiogram
│   (handlers.py)     │
└────────┬────────────┘
         │ HTTP (FastAPI)
         ▼
┌─────────────────────────────────────────┐
│   Оркестратор (FastAPI)                 │
│   - process_message (orchestrator.py)   │
│   - classify_request (ML + правила)     │
│   - история диалогов (PostgreSQL)       │
│   - суммаризация (summarizer.py)        │
│   - WebSocket-менеджер (ws_server.py)   │
└────────┬──────────────┬────────────────┘
         │ WebSocket     │ (локально, порт 8001)
         ▼               ▼
┌─────────────────┐   ┌──────────────────┐
│ Chrome Extension│   │  PostgreSQL       │
│ (content.js)    │   └──────────────────┘
│ - PoW (Worker)  │
│ - SSE-парсер    │
│ - Загрузка файлов│
└────────┬────────┘
         │ HTTPS
         ▼
┌─────────────────────┐
│ chat.deepseek.com   │
│ (веб-версия)        │
└─────────────────────┘
```

**Поток запроса:**
1. Пользователь отправляет текст/фото/файл в Telegram.
2. Бот сохраняет сообщение в БД.
3. Классификатор определяет режим (`instant`, `expert`, `vision`).
4. Формируется контекст (полная история + суммаризация).
5. Запрос через WebSocket отправляется в расширение Chrome.
6. Расширение:
   - Запрашивает PoW (Web Worker с WASM).
   - Получает `hif-leim` и `hif-dliq`.
   - Создаёт новую сессию DeepSeek.
   - Загружает файлы (при необходимости).
   - Отправляет запрос в `api/v0/chat/completion`.
   - Парсит SSE-поток и возвращает ответ.
7. Бэкенд сохраняет ответ ассистента и возвращает пользователю.

2. Включите «Режим разработчика»
3. Нажмите «Загрузить распакованное» и выберите папку `chrome-extension`

- Бот должен ответить на `/start`
- Healthcheck: `curl http://localhost:8001/health`
- Логи: `docker compose logs -f backend`

## Компоненты системы

### Telegram-бот

Обработчики сообщений находятся в `backend/telegram_bot/handlers.py`:
- `/start` — приветствие и справка
- `/new` — начать новый диалог (создаётся новая беседа в БД)
- **Текстовые сообщения** — автоматическая классификация и ответ
- **Фото и документы** — загрузка в DeepSeek с Vision
- **Альбомы** — до 10 файлов в одном запросе (собираются за 10 секунд)

### Бэкенд (FastAPI)

Основные компоненты:
- `main.py` — запуск сервера, WebSocket и бота
- `ws_server.py` — менеджер WebSocket-соединений с расширением
- `orchestrator.py` — обработка запросов, выбор режима, взаимодействие с расширением
- `summarizer.py` — асинхронная суммаризация длинных диалогов
- `models.py` — модели SQLAlchemy (User, Conversation, Message, ConversationSummary)

### Chrome-расширение

Файл `content.js` (внедряется на `chat.deepseek.com`):
- **WebSocket-клиент** для связи с бэкендом
- **PoW-генератор** через Web Worker и WASM-модуль `37627.ebf6d8f55d.js`
- **Загрузка файлов** через `api/v0/file/upload_file`
- **Парсинг SSE** (Server-Sent Events) для сбора ответа
- **Создание сессии** через `api/v0/chat_session/create` (каждый запрос в новом чате)

### Оркестратор и классификация режимов

Классификация запроса (`classify_request` в `orchestrator.py`):
1. **ML-классификация** (DeepSeek Instant) — приоритетный метод
2. **Правила** — если ML не сработал:
   - `instant`: поисковые маркеры (цена, погода, курс...)
   - `expert`: сложные запросы (напиши код, объясни...), длина > 500 символов
3. **По умолчанию** — `instant`

Режимы работы:
| Режим    | `model_type` | `search_enabled` | `thinking_enabled` | Модель             |
|----------|--------------|------------------|--------------------|--------------------|
| Instant  | `null`       | `true`           | `false`            | `deepseek-chat`    |
| Expert   | `"expert"`   | `false`          | `false`            | `deepseek-reasoner`|
| Vision   | `null`       | `false`          | `false`            | авто (файлы/фото)  |

**Запрос:**
```json
{
  "type": "REQUEST",
  "requestId": "uuid",
  "payload": {
    "prompt": "текст",
    "model_type": null,
    "search_enabled": true,
    "images": [{"base64": "data:mime;base64,...", "name": "file.png"}]
  }
}
```

**Ответ:**
```json
{
  "type": "RESPONSE",
  "requestId": "uuid",
  "result": {
    "success": true,
    "content": "ответ DeepSeek"
  }
}
```

### Запросы к DeepSeek (из расширения)
**Создание сессии:**
```http
POST /api/v0/chat_session/create
Content-Type: application/json
{}
```

**Загрузка файла:**
```http
POST /api/v0/file/upload_file
Content-Type: multipart/form-data
x-ds-pow-response: ...
x-model-type: vision
```

**Запрос completion:**
```json
{
  "chat_session_id": "...",
  "parent_message_id": null,
  "model_type": null,
  "prompt": "текст",
  "ref_file_ids": ["file-..."],
  "thinking_enabled": false,
  "search_enabled": true

```

Токены приходят в сообщениях вида:
```json

```

## Proof-of-Work (PoW)

DeepSeek требует решения криптографической задачи перед каждым запросом. Реализация:
1. Запрос challenge: `POST /api/v0/chat/create_pow_challenge`
2. Решение через **Web Worker** с WebAssembly (`wasm_solve`)
   - Алгоритм: `DeepSeekHashV1`
   - Критерий: `SHA3-256(challenge + salt + answer) < (2^256 / difficulty)`
   - `signature` возвращается сервером и передаётся обратно без изменений
3. Заголовок `x-ds-pow-response` формируется как base64 от:
```json
{
  "algorithm": "DeepSeekHashV1",
  "challenge": "...",
  "salt": "...",
  "answer": 12345,
  "signature": "...",
  "target_path": "/api/v0/chat/completion"
}
```

**Особенности:**
- PoW действителен 300 секунд (5 минут)
- Для загрузки файлов используется отдельный PoW с `target_path: "/api/v0/file/upload_file"`
- Каждый файл требует свежий PoW (нельзя переиспользовать)

## Vision и работа с файлами

**Поддерживаемые форматы:** практически все (PDF, DOCX, TXT, CSV, ZIP, изображения, код, логи и т.д. — полный список из 500+ расширений).

**Процесс обработки файла:**
1. Файл сохраняется локально в `uploads/`
2. Конвертируется в base64 с MIME-типом
3. Расширение загружает его через `upload_file` с PoW
4. Статус проверяется через `fetch_files` до `SUCCESS`
5. `ref_file_ids` передаются в основной запрос
6. Запрос отправляется с `x-model-type: vision`

**Ограничения:**
- Максимум 50 файлов в одном запросе
- Каждый файл до 100 МБ
- Альбомы Telegram — до 10 файлов

## Альбомы (Media Groups)

Telegram отправляет несколько фото/файлов как альбом. Логика обработки:
- Все вложения в рамках одного `media_group_id` собираются в список
- Таймер 10 секунд ожидает последний файл
- Затем все файлы отправляются одним запросом в `process_message` как `file_paths`
- Подпись берётся из первого сообщения с текстом

## База данных и история диалогов

**Схема БД:**
- `users` — пользователи Telegram
- `conversations` — беседы (одна активная на пользователя)
- `messages` — сообщения (роль, текст, путь к файлу, количество токенов)
- `conversation_summaries` — сжатые резюме для длинных диалогов

**Суммаризация:**
- Включается, когда количество несуммаризованных сообщений > `SUMMARIZE_AFTER_MESSAGES` (20)
- Выполняется асинхронно, не блокируя ответ
- Сжатый контекст вставляется как системное сообщение

**Очистка:** команда `/new` создаёт новую беседу, старая сохраняется в БД.

## Безопасность и ограничения

- **Аутентификация WebSocket:** ключ `supersecretkey` в query-параметре (заменить в production)
- **Авторизация:** используется токен пользователя из `localStorage` DeepSeek (сессия браузера)
- **Хранение файлов:** локально в `uploads/` (можно переключить на S3)
- **Ограничения Telegram:** до 10 файлов в альбоме, каждый до 50 МБ (сжатие фото)
- **CORS:** расширение работает в контексте страницы, ограничений нет
